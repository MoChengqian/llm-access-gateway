# SSE Streaming Format

Use this page after [Local Development](../local-development.md) and the first
stream check in [API Reference](../api.md). Once
`POST /v1/chat/completions` with `"stream": true` is returning bytes, this file
explains the exact SSE contract, the fallback boundary, and how to verify
mid-stream behavior without guessing.

## Fast Detailed Verification Path

If you only want the shortest streaming-specific proof path, run:

```bash
curl -i -N http://127.0.0.1:8080/v1/chat/completions \
  -H 'Authorization: Bearer lag-local-dev-key' \
  -H 'Content-Type: application/json' \
  -d '{"messages":[{"role":"user","content":"hello"}],"stream":true}'
./scripts/provider-fallback-drill.sh stream-fail
curl -s http://127.0.0.1:8080/metrics | grep -E 'lag_stream_(requests_total|chunks_total|ttft_milliseconds_)'
```

That path proves:

- streaming opens as SSE rather than JSON
- a healthy full stream ends with `data: [DONE]`
- a pre-first-chunk upstream failure can still fall back to another backend
- stream request, chunk, and TTFT metrics are exposed for correlation

To inspect the post-first-chunk failure boundary, read
[Streaming Failure Drill](../verification/failure-drills/streaming-failures.md)
in parallel. The gateway deliberately does not emit a fake `[DONE]` once bytes
have already been sent.

## Boundary At A Glance

The streaming contract has one critical split:

- before the first chunk is flushed, the gateway can still retry or fall back
- after the first chunk is flushed, the response is committed and can only finish or terminate

Everything below expands that boundary in protocol detail.

## Overview

When `stream: true` is set in a chat completion request, the gateway returns a stream of Server-Sent Events (SSE) instead of a single JSON response. This allows clients to receive and display tokens as they are generated by the model provider, reducing perceived latency.

## Content Type

Streaming responses use the `text/event-stream` content type with the following headers:

```
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
```

## SSE Event Format

Each event in the stream follows the SSE specification format:

```
data: <JSON payload>

```

Note the double newline (`\n\n`) after each event, which is required by the SSE specification.

## Chunk Structure

Each streaming chunk contains a JSON object with the following structure:

```json
{
  "id": "chatcmpl-123",
  "object": "chat.completion.chunk",
  "created": 1677652288,
  "model": "gpt-4o-mini",
  "choices": [
    {
      "index": 0,
      "delta": {
        "role": "assistant",
        "content": "Hello"
      },
      "finish_reason": null
    }
  ]
}
```

### Fields

- `id`: Unique identifier for this completion
- `object`: Always `"chat.completion.chunk"` for streaming responses
- `created`: Unix timestamp when the completion was created
- `model`: The model used for generation
- `choices`: Array of completion choices (typically one element)
  - `index`: Choice index (0 for single completion)
  - `delta`: Incremental content for this chunk
    - `role`: Role of the message (present in first chunk)
    - `content`: Text content fragment for this chunk
  - `finish_reason`: Reason for completion ending (null until final chunk, then "stop", "length", etc.)

## Stream Termination

### Normal Completion

When the stream completes successfully, the gateway sends a final marker:

```
data: [DONE]

```

This `[DONE]` marker indicates that no more chunks will be sent and the stream is complete.

### Example Complete Stream

```
data: {"id":"chatcmpl-123","object":"chat.completion.chunk","created":1677652288,"model":"gpt-4o-mini","choices":[{"index":0,"delta":{"role":"assistant","content":""},"finish_reason":null}]}

data: {"id":"chatcmpl-123","object":"chat.completion.chunk","created":1677652288,"model":"gpt-4o-mini","choices":[{"index":0,"delta":{"content":"Hello"},"finish_reason":null}]}

data: {"id":"chatcmpl-123","object":"chat.completion.chunk","created":1677652288,"model":"gpt-4o-mini","choices":[{"index":0,"delta":{"content":" there"},"finish_reason":null}]}

data: {"id":"chatcmpl-123","object":"chat.completion.chunk","created":1677652288,"model":"gpt-4o-mini","choices":[{"index":0,"delta":{"content":"!"},"finish_reason":null}]}

data: {"id":"chatcmpl-123","object":"chat.completion.chunk","created":1677652288,"model":"gpt-4o-mini","choices":[{"index":0,"delta":{},"finish_reason":"stop"}]}

data: [DONE]

```

### Failure Scenarios

If an error occurs **before the first chunk is written**, the gateway can attempt fallback to another provider. In this case, the client receives a standard HTTP error response instead of a stream.

If an error occurs **after the first chunk is written**, the gateway cannot retry or fallback because the HTTP response headers and partial content have already been sent to the client. In this case:

1. The stream is closed immediately
2. No `[DONE]` marker is sent
3. The client should detect the incomplete stream (missing `[DONE]` marker)

This constraint is fundamental to HTTP streaming: once headers and content have been flushed to the client, the response cannot be "taken back" or retried.

## Time to First Token (TTFT)

The gateway measures Time to First Token (TTFT) as the duration from when the streaming request begins until the first chunk is successfully written to the client.

TTFT is an important performance metric for streaming responses because it represents the perceived latency before the user sees any output.

### TTFT Measurement

The gateway tracks TTFT internally and exposes it via Prometheus metrics:

```
lag_stream_ttft_milliseconds_sum
lag_stream_ttft_milliseconds_count
```

These metrics allow you to calculate average TTFT and monitor streaming performance over time.

### What Affects TTFT

TTFT includes:
- Network latency to the upstream provider
- Provider's time to generate the first token
- Gateway processing overhead (auth, routing, governance)

TTFT does not include:
- Time to complete the entire stream
- Client-side rendering time

## Client Implementation

### Parsing SSE Streams

Clients should:

1. Make a POST request with `stream: true`
2. Read the response as a stream of text
3. Split on `\n\n` to separate events
4. Parse each `data:` line as JSON (except `[DONE]`)
5. Accumulate `delta.content` values to build the complete message
6. Detect stream completion by the `[DONE]` marker

### Example with curl

```bash
curl -N http://127.0.0.1:8080/v1/chat/completions \
  -H 'Authorization: Bearer lag-local-dev-key' \
  -H 'Content-Type: application/json' \
  -d '{
    "messages": [{"role": "user", "content": "Say hello"}],
    "stream": true
  }'
```

The `-N` flag disables buffering, allowing you to see chunks as they arrive.

### Example with JavaScript

```javascript
const response = await fetch('http://127.0.0.1:8080/v1/chat/completions', {
  method: 'POST',
  headers: {
    'Authorization': 'Bearer lag-local-dev-key',
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    messages: [{role: 'user', content: 'Say hello'}],
    stream: true
  })
});

const reader = response.body.getReader();
const decoder = new TextDecoder();
let buffer = '';

while (true) {
  const {done, value} = await reader.read();
  if (done) break;
  
  buffer += decoder.decode(value, {stream: true});
  const lines = buffer.split('\n\n');
  buffer = lines.pop(); // Keep incomplete line in buffer
  
  for (const line of lines) {
    if (line.startsWith('data: ')) {
      const data = line.slice(6);
      if (data === '[DONE]') {
        console.log('Stream complete');
        break;
      }
      const chunk = JSON.parse(data);
      const content = chunk.choices[0]?.delta?.content;
      if (content) {
        console.log(content);
      }
    }
  }
}
```

## Implementation Details

### Flushing Behavior

The gateway explicitly flushes each chunk to the client immediately after writing it. This ensures low latency and prevents buffering delays.

From `internal/api/handlers/chat.go`:

```go
flusher, ok := w.(http.Flusher)
if !ok {
    // Streaming not supported
    return
}

// Write chunk
fmt.Fprintf(w, "data: %s\n\n", payload)

// Flush immediately
flusher.Flush()
```

### Fallback Constraint

The gateway implements provider fallback for resilience, but streaming imposes a critical constraint:

**Fallback is only possible before the first chunk is written.**

Once the first chunk is flushed to the client:
- HTTP response headers have been sent (200 OK, text/event-stream)
- Partial content has been transmitted
- The response cannot be "undone" or retried

This is why the router in `internal/provider/router/chat.go` waits for the first event before considering a streaming request successful:

```go
firstEvent, err := p.awaitFirstStreamEvent(attemptCtx, events)
if err != nil {
    // Can still try next provider
    continue
}

// First chunk received - commit to this provider
return p.wrapStream(ctx, events, firstEvent, ...), nil
```

### Governance and Streaming

Governance checks (auth, rate limits, quotas) are performed **before** the stream begins. This ensures:

1. Invalid requests are rejected with proper HTTP error codes
2. Quota violations don't result in partial streams
3. Usage tracking captures the complete request

Token counting for streaming requests happens as chunks arrive, and the final usage is recorded after the stream completes.
